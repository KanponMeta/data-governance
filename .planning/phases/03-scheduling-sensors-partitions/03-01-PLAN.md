# Phase 3 计划 01：调度器 + 传感器 + 分区架构

## 目标

在 Phase 2 执行引擎之上交付自驱动子系统：cron 调度器守护进程、传感器评估循环、分区策略，以及回填优先级隔离。所有组件通过共享 tick 循环和 `SELECT FOR UPDATE SKIP LOCKED` 原语在多副本间安全运行。

## 架构概览

### 调度器守护进程

调度器作为新的 `./platform scheduler` 子命令运行（扩展 Phase 2 D-02 多模式模式）。调度器将运行直接插入 `runs` 表（不通过 ClaimNext）；worker 通过 ClaimNext 声明它们。调度器关闭 ⇒ 没有新运行排队，但进行中的运行不受影响。

**状态持久化是惰性的**：新的 `schedules` 表保存 `(asset_name, cron_expr, last_fire_at, next_fire_at, paused_at)`。每个 tick（默认 30s）：

1. `SELECT id, asset_name, cron_expr, next_fire_at FROM schedules WHERE next_fire_at <= NOW() FOR UPDATE SKIP LOCKED`
2. 为每行计算下一个 `next_fire_at`（使用 `robfig/cron/v3.Next()`）
3. 插入 `runs(asset_name, priority='normal', partition_key=...)` 行
4. `UPDATE schedules SET last_fire_at=NOW(), next_fire_at=<computed> WHERE id=$1`

`runs` 表**仅保存可声明的运行**，从不是未来运行。`next_fire_at` 是调度器行的属性，不是 runs 行的属性。

**多副本安全**：所有调度火灾使用 `FOR UPDATE SKIP LOCKED` — 两个调度器副本不会为同一调度器行重复火灾。

### 传感器模型

传感器运行在**相同的 `scheduler` 子命令**内，共享 tick 循环和 SKIP LOCKED 原语。新 `sensors` 表：

```sql
CREATE TABLE sensors (
    id UUID PRIMARY KEY,
    asset_name TEXT NOT NULL,
    sensor_name TEXT NOT NULL,
    min_interval_seconds INTEGER NOT NULL DEFAULT 30,
    last_evaluated_at TIMESTAMPTZ,
    last_fired_at TIMESTAMPTZ,
    last_run_key TEXT,
    cooldown_until TIMESTAMPTZ,
    consecutive_failures INTEGER NOT NULL DEFAULT 0,
    disabled_at TIMESTAMPTZ,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE (asset_name, sensor_name)
);
```

每个 tick：
1. `SELECT id, asset_name, sensor_name, min_interval_seconds FROM sensors WHERE disabled_at IS NULL AND (last_evaluated_at IS NULL OR last_evaluated_at <= NOW() - min_interval) FOR UPDATE SKIP LOCKED`
2. 对于每行：调用用户的 `Sense(ctx)` 函数
3. 两层去重检查：
   - RunKey 与 `last_run_key` 比较（相同 ⇒ 跳过）
   - `NOW() < cooldown_until`（在冷却中 ⇒ 跳过）
4. 如果应触发：插入 `runs(asset_name, priority='normal')` 行 + 更新 `last_fired_at` 和 `last_run_key`
5. 如果 `Sense()` 返回错误：记录 + `consecutive_failures++`；如果 `consecutive_failures >= 60`，设置 `disabled_at=NOW()`

### 分区模型

**DSL**：`.Partitions(spec)` 方法，spec 实现 `asset.PartitionStrategy` 接口：

```go
type PartitionStrategy interface {
    WindowKey(when time.Time) string  // 返回 ISO-8601 键
    ParseKey(key string) (interface{}, error)
}

type DailyPartitions struct{ Start time.Time; TZ *time.Location }
type WeeklyPartitions struct{ Start time.Time; TZ *time.Location }
type MonthlyPartitions struct{ Start time.Time; TZ *time.Location }
type CategoryPartitions struct{ Keys []string }
```

每个资产最多一个分区策略。`runs.partition_key` 列存储分区键；非分区资产该列为 NULL。唯一约束 `(asset_name, partition_key)` 作用域为进行中状态，防止同一分区的重复并发运行。

### 回填优先级隔离

**三层隔离**（PITFALLS #6 要求）：

1. **优先级枚举**：`runs.priority` — `'critical' | 'normal' | 'backfill'`（默认 `'normal'`）
2. **优先级 FIFO 声明**：`ClaimNext ORDER BY CASE priority WHEN 'critical' THEN 0 WHEN 'normal' THEN 1 WHEN 'backfill' THEN 2 END ASC, queued_at ASC`
3. **并发 token 池资源标签**：回填运行额外获取 `backfill` 标签的 token（Phase 2 D-16 池）

```go
// executor.Run 中的回填优先级处理
if claimed.Priority == PriorityBackfill {
    _, err := e.deps.Pool.Acquire(ctx, PoolAcquireRequest{
        ResourceTag: "backfill",
        Weight:     1,
    })
    // 如果获取失败：释放所有 acquired tokens 并返回错误（run 保持 queued）
}
```

## 威胁模型

| 威胁 | 处理 | 原因 |
|------|------|------|
| 调度火灾重复（多副本） | Mitigation | `FOR UPDATE SKIP LOCKED` 防止重复火灾 |
| 传感器评估过度（RunKey 碰撞） | Mitigation | 两层去重（RunKey + cooldown）|
| 传感器 `Sense()` 长时间阻塞 | Accept | 传感器在共享 tick 中运行；如果 `Sense()` 慢，它延迟后续传感器 |
| 回填饥饿正常运行 | Mitigation | 优先级排序 + `backfill` token 资源标签 |
| 分区键冲突 | Mitigation | 唯一约束 `(asset_name, partition_key)` 作用域为进行中状态 |
