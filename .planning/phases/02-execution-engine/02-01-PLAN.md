# Phase 2 Plan 03: 重试引擎 + 全局并发令牌池 + 连接器配置 + 运行执行器 - 概要

端到端运行执行器，集成重试、并发、配置和每个运行的 heartbeat goroutine — 将 plan 02-01 和 02-02 整合为完整的执行引擎。

## 已构建内容

### 任务 1: concurrency_tokens 表 + Pool + retry/policy + retry_scheduled 事件

**ConcurrencyToken ent 实体** (`internal/storage/ent/schema/concurrency_token.go`):
- 字段: id, run_id, asset_name, resource_tag, weight, acquired_at
- 索引: resource_tag (容量检查), run_id (清理), acquired_at (过期检测)
- 所有并发限制使用单一表 (D-16 不可协商): run级别、op级别、resource级别

**迁移** (`migrations/20260507121500_phase2_concurrency_tokens.sql`):
- 手工编写 SQL (网络隔离环境中 atlas-provider-ent 不可用)
- Platform_owner 所有权 + platform_app DML 权限，与 Phase 1 模式一致

**Pool** (`internal/concurrency/pool.go`):
- `Acquire(ctx, runID, assetName, tag, weight)`: BEGIN → SELECT SUM(weight) FOR UPDATE → 检查限制 → INSERT → COMMIT。如果超过限制则返回 ErrCapacity。
- `Release(ctx, runID, tag)`: DELETE WHERE run_id=$1 AND resource_tag=$2。幂等。
- `ReleaseAll(ctx, runID)`: 删除某个运行的所有令牌。在终止状态时调用。
- `ReleaseStale(ctx, staleAfter)`: DELETE WHERE acquired_at < cutoff。在 worker 启动时调用 (plan 02-04)。
- 未配置的 tag 获得限制 = 2^31-1 (无限)，但仍记录令牌行以供可观察性。

**重试策略** (`internal/retry/policy.go`):
- `Schedule(attempt, policy)`: `delay = InitialDelay * 2^(attempt-1)`，上限为 MaxDelay，然后通过 `math/rand/v2` 添加抖动
- `ShouldRetry(attempt, policy)`: 返回 `attempt < policy.Max`; 零策略始终返回 false
- Attempt 从 1 开始索引: attempt=1 使用 InitialDelay, attempt=2 使用 2×InitialDelay

**run.step.retry_scheduled 事件** (D-18 完成):
- `EventTypeRunStepRetryScheduled = "run.step.retry_scheduled"` 添加到 AllKnownTypes()
- `RunStepRetryScheduledPayload{AssetName, Attempt, NextAttempt, ScheduledAt, DelayMs, Error}`
- 在延迟 sleep **之前**写入 event_log — 即使 worker 在延迟 sleep 期间崩溃，审计追踪仍然存在

### 任务 2: 连接器配置加载器 + 运行执行器 + heartbeat goroutine

**配置加载器** (`internal/connector/config/config.go`):
- `Load(yamlBytes []byte) (*Config, error)`: 在 yaml.Unmarshal **之前**解析 raw bytes 中的 ${ENV_VAR}
- `resolveEnv()`: `regexp.MustCompile(`\$\{([A-Z_][A-Z0-9_]*)\}`)` 模式，返回缺失的变量名 (从不返回值)
- 敏感值永不记录 — 测试 4 (TestLoad_DoesNotLogSecrets) 通过 slog 捕获断言
- 任何未解析的占位符返回 `ErrMissingEnvVar`，包装所有缺失的变量名
- `LoadFile(path)` 从磁盘读取; PLATFORM_CONFIG 环境变量约定已文档化

**FactoryRegistry** (`internal/connector/config/resolver.go`):
- `Factory func(params map[string]interface{}) (connector.Connector, error)` — plan 02-04 实现 postgres factory
- `BuildAll(cfg *Config, reg *connector.Registry)` — 遍历连接器，调用 factory，调用 RegisterInProcess
- 未知连接器类型返回格式化错误; 注册失败向上传播

**Executor** (`internal/runtime/executor.go`):

```
Executor.Run(ctx, runID, assetName):
  ┌─ spawn heartbeatLoop goroutine (immediate first tick)
  ├─ Registry.Get(assetName) → build transitive subgraph → TopologicalOrder()
  ├─ transition starting → running → emit run.started
  ├─ for each asset in topological order:
  │   ├─ acquire global token + per-resource tokens (single pool, D-16)
  │   ├─ emit run.step.started
  │   ├─ safeMaterialize() [defer/recover wraps MaterializeFunc]
  │   ├─ release tokens
  │   ├─ on success: emit run.step.succeeded → continue
  │   └─ on failure: emit run.step.failed
  │       ├─ if ShouldRetry: scheduleRetry() → emit retry_scheduled → sleep → retry
  │       └─ else: fail run (emit run.failed, ReleaseAll, return error)
  └─ on all steps ok: transition running → succeeded → emit run.succeeded → ReleaseAll
  defer: hbCancel() + hbWG.Wait() [goroutine MUST exit before Run returns]
```

**heartbeatLoop** goroutine:
- 每 HeartbeatInterval (默认 30s) tick 一次 `runs.last_heartbeat = NOW()`
- 第一次 tick 立即执行，这样即使非常短的运行也会留下 heartbeat 追踪
- 当 hbCtx 取消时退出 (在 Run 的 defer block 中取消)
- sync.WaitGroup 确保 goroutine 在 Run 返回前完全退出 (T-02-03-10)
- 10x 安全边际: 30s tick vs 5m reaper 阈值

## 双层故障模型 (D-14 Option B)

| 故障类型 | 处理器 | 机制 |
|---|---|---|
| 业务故障 (MaterializeFunc 返回错误) | 引擎重试循环 | retry.Schedule + retry.ShouldRetry; 在同一 Executor.Run 内inline重新执行 |
| 基础设施故障 (worker 崩溃、OOM、missed heartbeat) | Plan 02-04 stale-run reaper | 扫描 `last_heartbeat < NOW() - 5m`，调用 TransitionForReset → 重新排队 |

River.max_attempts 接线明确根据 D-14 Option B 推迟到后续阶段。Reaper (plan 02-04) 是 Phase 2 的规范崩溃恢复机制。
