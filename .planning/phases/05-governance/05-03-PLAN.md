# Phase 5 Plan 05-03: PII 传播 + 管道内掩码

**同步 PII 标签传播：沿 column_edges 在 lineage_writer 事务内传播，下游列零未掩码窗口。`Builder.Column().TagOverride()` 是移除继承 pii=true 标签的唯一合规路径（首次写入 metadata.tag_overridden，幂等重复运行）。非仓库连接器（Postgres/MySQL/S3/GCS/HDFS）的 AssetIO.Write 由 MaskingIO 装饰，在管道内应用 HMAC-SHA256/redact/partial 转换。**

## 依赖关系

- 依赖 Phase 05-01（审计哈希链）+ Phase 05-02（column_policies + enforcement_mode）
- 提供 RBAC-05（in-pipeline 掩码）
- 影响 Phase 05-04（治理审批工作流）、Phase 06（可观测性）

## 任务

### Task 1: PII 传播器（同步、union 规则、override 感知）+ TagOverride Builder DSL + lineage_writer 钩子

**行为：**
- **算法：** 对每个输出列：(1) 若 asset_metadata 行有 `pii_override` 标签且非空 reason → 跳过（override 优先）；(2) 否则 SELECT EXISTS（任何上游列携带 `tags ? 'pii'`）→ 为真则 UPSERT `asset_metadata.tags = tags || '{"pii":true}'`。(3) 每次新的 pii=true 赋值不产生新的 audit 条目（传播是正常物化的副作用）。TagOverride 移除 pii 才产生 `metadata.tag_overridden`。
- **Builder DSL** 链式方法在 `*ColumnBuilder` 上：`asset.New("orders_anon").Column("hashed_ssn").TagOverride(asset.TagOverride{Remove: "pii", Reason: "hashed at source"})`
- **TagOverride struct**: `{Remove string; Add string; Reason string}`。Remove 或 Add 必须非空；Reason 必须非空。
- **lineage.Writer.CaptureRun 钩子：** 在 column_edges UPSERT 后、tx 提交前，调用 `propagator.Propagate(ctx, tx, runID, outputColumnRefs)`
- **同一事务保证：** 传播器必须在调用方的 `*sql.Tx` 内运行，不开新连接或新事务。

**操作：**

### Task 2: 管道内掩码函数 + MaskingIO 装饰器 + 非仓库连接器 executor 接线（RBAC-05）

**行为：**
- **掩码函数**（纯 Go，确定性）：
  - `MaskHash(value, salt)` — `hex(HMAC-SHA256(salt, value))`。Salt 来自 `MASK_HASH_SALT` env var；生产模式（GOV_ENV=prod）要求非空 salt。
  - `MaskRedact(value)` — 返回 `"***"`
  - `MaskPartial(value, reveal)` — 若 `len(value) <= 2*reveal+1` 返回 redact；否则返回 `value[:reveal] + strings.Repeat("*", len(value)-2*reveal) + value[len(value)-reveal:]`
- **MaskingIO 装饰器：** 包装 AssetIO。其 Write 读取每行，查找目标资产的活动列策略，为 pii=true 或有 in-pipeline 策略的列应用相应掩码函数。
- **executor.runStep 接线** (D-05 能力断言顺序)：
  1. 解析资产的连接器
  2. 若连接器实现 `connector.MaskingProvisioner` → 掩码是仓库原生的，不包装 MaskingIO
  3. 否则（非仓库）→ 检查是否有列满足 `tags ? 'pii'` 或 `column_policies.enforcement_mode='in-pipeline'` → 有则用 `NewMaskingIO` 包装
  4. 否则 → 不包装（原 Phase 4 路径不变）
- **pii 无策略时 fallback：** slog.Warn + 应用 redact 作为安全默认值

**操作：**
1. 创建 `internal/policy/mask.go`（纯 Go 掩码函数 + Salt() 记忆化读取）
2. 创建 `internal/policy/mask_test.go`（10 个单元测试）
3. 创建 `internal/asset/io_masking.go`（MaskingIO 装饰器 + MaskRule）
4. 创建 `internal/asset/io_masking_test.go`（10 个单元测试，含并发测试）
5. 扩展 `internal/runtime/executor.go`（MaskRule + MaskRulesProvider + maybeWrapMaskingIO）
6. 扩展 `internal/policy/store.go`（MaskRulesForAsset 方法）
7. 创建 `internal/runtime/executor_mask_test.go`（9 个单元测试）
8. 测试 + 提交

## 偏差

1. **column_pii_tags 新表引入** — asset_metadata 是 APPEND-ONLY + FORCE ROW LEVEL SECURITY，Phase 4 锁死了 UPSERT 路径。新建专表，PRIMARY KEY (asset, column_name)，propagator 的 UPSERT 定义明确。
2. **迁移文件名 20260510000004** — 协调器分配，与 plan 05-02 和 plan 05-05 并行。
3. **lineage.Writer 保持 2 参数构造函数** — PII 传播通过 `WithPropagator(*Propagator)` 灵活设定器实现，完全向后兼容。
4. **MaskingIO 使用 MaskApplyFunc 依赖注入** — internal/policy 已导入 internal/asset，直接导入 policy 会产生循环依赖。Executor 在调用站点注入 policypkg.Apply。
5. **runtime.MaskRulesProvider 返回 []policy.MaskRule** — 保持 policy 类型在 policy 包内，runtime 通过接口依赖。
6. **maybeWrapMaskingIO 提取为 export_test.go 导出的辅助函数** — 四个命名验收标准测试需要无 DATABASE_URL 的纯单元测试。
