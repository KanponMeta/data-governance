Task 1 ships the 6 ent entities + Atlas-generated migration. Task 2 appends the hand-managed appendix (RLS, CHECK constraints, partial indices, event_log enum extension). **Schema struct placement (Pitfall 4 resolution):** The connector package already exports `Column` and `SchemaResponse`. Wave 1 promotes the existing thin `Column` to a richer struct with `Default`, `IsPrimaryKey`, `Comment`, and `Type` (normalized) fields, and adds a new `Schema` struct with `PrimaryKey`, `RowCountEstim`, `CapturedAt`. The legacy `SchemaResponse` is kept (don't break Phase 1 contract) and Wave 2 will route consumers from `Schema()` (returns `SchemaResponse`) to the new `DescribeSchema()` (returns `Schema`).

**RLS for schema_changes (Open Question 3 resolution):** 04-RESEARCH.md Open Question 3 recommends "allow UPDATE on schema_changes for platform_app, restricted by application logic" rather than column-level RLS. This plan follows that recommendation — DB grants `SELECT, INSERT, UPDATE` (no DELETE/TRUNCATE), and the Wave 6 ent mutation is responsible for only setting the ack columns. Test `TestSchemaChangesAckOnly` (Wave 6) will assert that only the three ack columns can be modified by the platform_app ent client. Acknowledgement is auditable because the ent mutation logs an event_log row (D-21 `schema.break_acknowledged`).

**Optional `runs.code_hash` and `runs.lineage_emitted` columns:** 04-RESEARCH.md §3 explicitly states "Lineage_emitted column on runs: NOT needed if capture is in the same tx." This plan therefore does NOT add those columns. If Wave 3 finds the same-tx approach insufficient (e.g., because executor refactor is too invasive), Wave 3 will add a follow-up migration. Avoiding speculative columns keeps the schema minimal.
